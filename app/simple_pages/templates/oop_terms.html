Namespace

A namespace is a collection of classes organized logically in the program.
Namespaces help define the scope of the names within a class.

Namespaces are like folders containing classes where no two classes can have the same name. They help logically organize the code.

In python, adding __init__.py to a directory makes classes in the folder exist inside the namespace.

We use this in our calculator many times. For example, when we include __init__.py in the operations folder, it becomes a namespace. The classes inside become part of the operations namespace.
The namespace for the __init__.py file is the name of the folder the namespace is in. __init__ is the default file that goes in the folder for the namespaces.

Instance

Instance Methods

Instance methods are methods that can be called on an instance of a class.
They operate on each individual object in memory.

Here is an example of an instance method in calculations/__init__.py:


The get_result() method is an instance method that operates on the self.values property of that particular Addition instance.

Whenever we see self as a parameter in a method, it is an instance method.

Instance Properties

Instance properties are properties that are encapsulated in an instance.
They are the properties of a class can be accessed from the instance of that class.

An example of an instance property can be found in our Addition class (see above) as self.values contains the values for each instance of the Addition class.

Each instance has its own independent values for their properties.

For example, if we were to instantiate an Addition instance with self.values = (1, 2) and another with self.values = (3, 4), and if we were to call get_result() on both, we would get 3 and 7, respectively.

Encapsulation

Encapsulation is the concept of wrapping attributes and behavior into a class.
When an object is instantiated, it encapsulates its variables and functions from the class into a single unit.

We see encapsulation in action when looking at instance methods and properties that come with an instance each time we instantiate a class.

For example, we see encapsulation when we instantiate the Addition class (see instantiation above).

To see examples and learn more about encapsulation, please refer to the OOP Principles article.

Inheritance

Inheritance is the concept of defining a class that inherits the data and behavior from another class.
The class that is being inherited from is called the parent or base class and the inheriting class is called the child class.

We see inheritance in action when our Addition, Subtraction, Multiplication, and Division classes inherit properties and methods from their Calculation parent class.

To see examples and learn more about inheritance, please refer to the OOP Principles article.



Polymorphism

Polymorphism is the concept of using a common interface for multiple forms of data types.
The class that is being inherited from is called the parent or base class and the inheriting class is called the child class.

We see polymorphism in action in our operation calculation classes of Addition, Subtraction, Multiplication, and Division in calculations/__init__.py.

Each operation calculation class inherits from the Calculation class and has a get_result() method that knows how to perform its calculation on the values for each different type of calculation with the same interface.

To see examples and learn more about polymorphism, please refer to the OOP Principles article.

Abstraction

Abstraction is the concept of hiding the internal implementations of a process or method from the user.
We see abstraction in action in our Calculator class with the add(), subtract(), multiply(), and divide() methods.

The actual code and methods that do the calculating are hidden in calculations/__init__.py and operations/__init__.py.

This simplifies the job of the user by allowing them only to call add() from the calculator interface to do their operations.

To see examples and learn more about abstraction, please refer to the OOP Principles article.



OVERVIEW OF SOLID

While object-oriented programming is defined by encapsulation, polymorphism, inheritance, and abstraction, SOLID acts as the rules for OOP.

SOLID mainly consists of 5 object-oriented design principles including: Single-Responsibility Principle, Open-Closed Principle, Liskov Substitution Principle, Interface Segregation Principle, and Dependency Inversion Principle.

To see examples and learn more about SOLID, please refer to the SOLID article.



Single-Responsibility Principle

The Single-Responsibility Principle states that a function should do one and only one thing.
There should be one and only place that controls a single action in a program.

Open-Closed Principle

The Open-Closed Principle states that a function should be open for extension but closed for modification.

Once a function is finished, you shouldn’t need to go back and change it for any reason. You can modify how it works by adding to it externally but the function itself shouldn’t need to be changed.

Liskov Substitution Principle

The Liskov Substitution Principle states that we should be able to substitute parent classes and child classes together.

Every subclass should be able to substitute for their parent class.

Interface Segregation Principle

The Interface Segregation Principle states that classes should be separated in a way that avoids having unused methods inherited in another class.

To avoid this, classes should do one and only one thing.

Dependency Inversion Principle

The Dependency Inversion Principle states that classes should depend on abstract classes, not concrete ones.

This helps decouple classes and allow them to call each other in a way that allows extensions rather than hardcoded dependencies.











OVERVIEW OF DESIGN PATTERNS

When you take the properties of OOP and combine them with SOLID, the result becomes design patterns.

Design patterns are established, consistent industry standard ways of doing 23 of the most common actions a program needs to do in OOP.

They help each programmer's solutions to look alike to more easily be able to pick up and work on someone else's code.

Some of the design patterns we use in our calculator include: facade, factory, command, and decorators.

To see examples and learn more about design patterns, please refer to the SOLID article.
